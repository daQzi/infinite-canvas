# Tauri 原生文件下载设计

## 背景

当前图片、视频、音频和 ZIP 导出统一使用浏览器库 `file-saver`。网页端可以依靠浏览器下载能力保存文件，但 Tauri WebView 不提供稳定且一致的浏览器下载流程，导致桌面应用点击下载后无法正常落盘。

本次改动为 Tauri 桌面端提供系统原生“另存为”能力，同时保持 Web/Docker 版本的下载行为不变。

## 目标

- 覆盖生图工作台、视频工作台、画布媒体、我的素材、素材 ZIP 和画布 ZIP 的全部下载入口。
- Tauri 端点击下载后弹出系统原生“另存为”对话框。
- Web/Docker 端继续使用 `file-saver`。
- 支持 HTTP/HTTPS 地址、应用内部媒体、Blob、Data URL 和前端生成的 ZIP。
- 用户取消保存时安静结束；保存成功或失败时给出明确提示。
- HTTP/HTTPS 大文件由 Rust 直接获取，避免完整文件在 Rust 和 WebView 之间往返传输。

## 非目标

- 不实现下载队列、断点续传或后台下载管理器。
- 不改变媒体生成、应用内媒体存储和 WebDAV 同步逻辑。
- 不允许前端任意指定本机写入路径；目标路径必须由原生保存对话框产生。

## 方案比较

### 前端直接使用 Tauri Dialog/FS 插件

实现较短，但文件读取、路径选择和写入逻辑仍由前端编排。大文件需要转换成前端字节数组，下载能力也容易分散到各页面。

### 统一前端入口与 Rust 下载后端（采用）

前端只描述文件来源和建议文件名，Rust 负责原生对话框、远程获取、本地媒体读取和写盘。该方案符合项目现有 `src-tauri/src/backend/` 边界，能统一错误处理，并可让远程大文件留在 Rust 层处理。

### 拦截 WebView 下载行为

页面改动较少，但 Blob、Data URL 和不同平台 WebView 的下载行为不一致，无法稳定覆盖当前所有文件来源。

## 架构

### 前端下载服务

新增一个纯下载服务，接收统一请求：

```ts
type DownloadSource =
    | { kind: "url"; url: string }
    | { kind: "storage"; storageKey: string }
    | { kind: "blob"; blob: Blob };

type DownloadResult = { status: "saved"; path?: string } | { status: "cancelled" };
```

服务根据运行环境分流：

- Web/Docker：把来源解析为 `Blob` 或 URL，继续调用 `file-saver`。
- Tauri：调用 Rust 下载命令。Blob 和 Data URL 在前端转换为 Base64；HTTP/HTTPS URL 和 `storageKey` 保持引用形式交给 Rust。

多个页面共用的成功、失败提示由全局下载 Hook 统一处理。取消保存不显示错误提示。

### Rust 下载模块

新增 `src-tauri/src/backend/download.rs` 并注册下载命令。命令只接收建议文件名和文件来源，不接收前端提供的目标路径。

处理顺序：

1. 校验并清理建议文件名，保留合理扩展名。
2. 打开系统原生“另存为”对话框。
3. 用户取消时返回 `cancelled`，不读取或下载文件。
4. 用户确认后按来源获取内容：
   - `url`：仅允许 HTTP/HTTPS，由 Rust HTTP 客户端获取。
   - `storage`：从应用媒体目录读取对应 `storageKey`，并继续使用现有安全文件名规则。
   - `base64`：解码前端传入的 Blob、Data URL 或 ZIP 内容。
5. 写入对话框返回的路径并返回 `saved`。

Rust 模块复用现有 HTTP 客户端和媒体目录能力，必要的内部读取函数调整为模块内可复用，不改变已有 Tauri 命令契约。

## 下载入口迁移

以下入口从直接调用 `saveAs` 改为统一下载服务：

- 生图工作台结果图片。
- 视频工作台结果视频。
- 画布图片、视频和音频节点。
- 我的素材中的图片和视频。
- 我的素材 ZIP 导出。
- 画布项目 ZIP 导出。

有 `storageKey` 时优先使用内部媒体来源；只有没有内部存储引用时才使用 URL 或 Blob。这样可避免 Blob URL 失效，也能减少不必要的数据转换。

## 文件名与格式

- 沿用各入口现有建议文件名和扩展名。
- 文件名中的路径分隔符和系统非法字符会被替换，防止建议文件名改变保存目录。
- 原生对话框默认展示建议文件名；用户仍可修改名称和目标目录。
- 不根据远程响应擅自覆盖用户确认后的文件名。

## 交互与错误处理

- 点击下载后只出现一个系统保存对话框。
- 用户取消：保持当前页面状态，不弹出错误。
- 保存成功：提示“文件已保存”，可附带实际路径。
- 来源不存在、远程请求失败、Base64 解码失败或写盘失败：返回中文错误，由前端统一提示。
- 重复点击产生独立保存操作，不引入全局下载队列。

## 安全约束

- 远程下载仅允许 `http` 和 `https`。
- `storageKey` 继续经过现有文件名清理，不能越出应用媒体目录。
- 最终写入位置只能来自系统原生对话框。
- Rust 错误信息不得包含请求认证头、API Key 或文件内容。

## 验证

- 前端单元测试：Web/Tauri 分流、三种来源载荷、取消与失败结果。
- Rust 单元测试：文件名清理、来源校验、Base64 解码和内部媒体路径约束。
- TypeScript 类型检查和 Tauri 前端构建。
- macOS 人工验证图片、视频、音频、素材 ZIP 和画布 ZIP 均能打开“另存为”并落盘；取消对话框不会报错。
- Windows 构建产物至少验证原生对话框、文件名和写盘行为。
